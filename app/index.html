The aim of this project was to abstract the creation of the Delivery Man System into decoupled, classes all responsible for their own behaviour.

The classes are;

  • TakeAwayTown, which in hindsight should be called Grid
  • Points, the creations of random unique points between 1 and the size of the grid.
  • Actors being any object placed on the grid and a reference to its position on the grid,
  • A class to calculate the nearest actor/actors to a nominated actor in this case the JEB.


Each class exists independently from each other.
  • So you can use the Grid (TakeAwayTown) class to create any grid.
  • This is also true for the points, which can be used to create any points within a given range.
  • The actors are created using a grid, points, and the name of the actors. But any data can be used to create it.

This keeps with the sprit of reusability. And is much better for change management since each class can be changed without effecting other class.

The whole system was created by Pivotal Labs Jasmine BDD which is based on Ruby's Rspec. The test suite ensure the code is working correctly and gives developer the confidence to alter code because the suite reports any error. This is evident in the project when I realised I need to get a hash of actors and their co-ordinated rather then their random points. I was able to alter my code in 5 mins knowing everything worked fine. As Uncle Bob say "Testing makes software soft again!"

In addition BDD made my ask very important questions regarding the design of the classes it also forced me to write clean, concise code!
